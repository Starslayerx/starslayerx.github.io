## Getting Started with pytest

这是一个基本测试：

```Python
# test_one.py
def test_passing():
    assert (1, 2, 3) == (1, 2, 3)
```

该函数以 `test_` 开头，会被 pytest 发现并运行。
`assert` 语句会决定测试通过或者失败，assert 是一个 Python 关键字。
若结果为 `False` 会抛出一个 `AssertionError` 异常，任何未捕获的异常都会导致测试失败。
尽管任何错误都会导致失败，但传统上仍然使用 `AssertionError` 断言异常来决定成功或失败。


**Test Discovery**

pytest 执行过程中，自动寻找并确定要运行哪些测试的部分被称为 test discovery 测试发现。
下面是测试代码的命名规范：

- 测试文件应该命名为 `test_<something>.py` 或 `<something>_test.py`
- 测试方法和函数应该命名为 `test_<something>`
- 测试类命名为 `Test<Something>`

**Test Outcomes**

目前已经有一个通过和失败的测试示例。
然而，pass 和 fail 并不是仅有的可能输出。

有这些所有可能输出：

- PASSED (.): 测试成功
- FILAED (F): 测试失败
- SKIPPED (S): 测试跳过，使用 `@pytest.mark.skip()` 或 `@pytest.mark.skipif()` 装饰器告诉 pytest 来跳过测试
- XFAIL (X): 不应该通过的测试，运行并失败了
- XPASS (X): 测试使用 xfail 标记过，但是运行通过了
- ERROR (E): 在夹具 fixture 或钩子 hook 函数执行期间发生了异常，而不是在测试函数执行期间

## Writing Test Functions

这节首先会介绍如何编写测试函数，以及如何组织测试类、模块和目录。

我们编写的测试代码需要能够运行应用代码，该应用代码是需要验证的代码。
它有很多名称，如 code under test (CUT), system under test (SUT), device under test (DUT)，这里统一使用 application code 应用代码。
讽刺的是，test code 通常是明确无误 unambiguous 的，并且没有很多名称。

### Writing Knowledgeable-Building Tests

这里要为 `cards` 这个代码库编写测试，首先要安装：

```console
$ pip install -e ./cards_proj
```

或者使用 uv 添加为依赖：

```console
$ uv add --editable ./cards_proj
```

该 Cards 源码分为三部分：CLI, API, 和 DB。
CLI 处理用户输入，调用 API 处理逻辑，然后再连接 DB 数据库。

```Python
# cards_proj/src/cards/api.py
@dataclass
class Card:
    summary: str = None
    owner: str = None
    state: str = "todo"
    id: int = field(default=None, compare=False)

    @classmethod
    def from_dict(cls, d):
        return Card(**d)

    def to_dict(self):
        return asdict(self)
```

`Card` 类拥有 `summary`, `owner` 和 `state` 这些字符串字段，和一个整数字段 `id`。
其通过 `field()` 方法设置默认值为 None，并通过 `compare=False` 实现不进行比较，即两个 `Card` 比较时不会对比 `id` 的值。还有两个方法，`from_dict` 和 `to_dict`。

当面对新的数据结构的时候，编写测试能帮助理解其如何工作。

```Python
# test_card.py
from cards import Card

def test_field_access():
    c = Card('something', 'brain', 'todo', 123)
    assert c.summary == 'something'
    assert c.owner == 'brain'
    assert c.state == 'todo'
    assert c.id == 123

def test_defaults():
    c = Card()
    assert c.summary is None
    assert c.owner is None
    assert c.state == 'todo'
    assert c.id is None

def test_equality():
    c1 = Card('something', 'brain', 'todo', 123)
    c2 = Card('something', 'brain', 'todo', 123)
    assert c1 == c2

def test_equality_with_diff_ids():
    c1 = Card('something', 'brain', 'todo', 123)
    c2 = Card('something', 'brain', 'todo', 4567)
    assert c1 == c2

def test_ineuqality():
    c1 = Card('something', 'brain', 'todo', 123)
    c2 = Card('completely different', 'okken', 'done', 123)

def test_from_dict():
    c1 = Card('something', 'brain', 'todo', 123)
    c2_dict = {
        'summary': 'something',
        'owner': 'brain',
        'state': 'todo',
        'id': 123,
    }
    c2 = Card.from_dict(c2_dict)
    assert c1 == c2

def test_to_dict():
    c1 = Card('something', 'brain', 'todo', 123)
    c2 = c1.to_dict()
    c2_excepted = {
        'summary': 'something',
        'owner': 'brain',
        'state': 'todo',
        'id': 123,
    }
    assert c2 == c2_excepted
```

运行测试

```console
$ uv add --dev pytest
$ uv run pytest ch2/test_card.py
```

### Failing with pytest.fail() and Exceptions

如果有未捕获异常，测试将失败：

- `assert` 时间失败，会报 `AssertionError` 错误
- 测试代码调用 `pytest.fail()`，抛出一个异常
- 或抛出任何其他异常

虽然任何异常都可能导致测试失败，但更加倾向于使用 `assert`。

### Writing Assertion Helper Functions

assertion helper 函数用于包装复杂的断言检查。
例如，Cards 数据类的设置是的两张具有不同 ID 的卡片仍会被报告为相等。
如果想要更加严格的检查，我们可以编写一个这样的 `assert_idential` 函数：

```Python
# ch2/test_helper.py
import pytest
from cards import Card

def assert_idential(c1: Card, c2: Card):
    __tracebackhide__ = True
    assert c1 == c2
    if c1.id != c2.id:
        pytest.fail(f"id's don't match. {c1.id} != {c2.id}")

def test_identical():
    c1 = Card('foo', id=123)
    c2 = Card('foo', id=123)
    assert_idential(c1, c2)

def test_idential_fail():
    c1 = Card('foo', id=123)
    c2 = Card('foo', id=456)
    assert_idential(c1, c2)
```

`assert_idential` 函数设置了 `__tracebackhide__=True`。
这个是可选的，这样失败测试不会在 traceback 中包含该函数。
普通的 `asssert c1 == c2` 用于检查除了 ID 外一切都相等。

要注意，断言重写 assert rewriting 仅应用于 `conftset.py` 文件和测试文件。

### Testing for Excepted Exceptions

如果某部分代码本来就会抛出异常，要如何测试呢？
可以使用 `pytest.raises()` 来测试预期的异常。

```Python
# ch2/test_exceptions.py
import pytest
import cards

def test_no_path_raises():
    with pytest.raises(TypeError):
        cards.CradsDB()
```

该语句 `with pytest.raises(TypeError)` 说无论下一个块的代码是什么，都应该抛出 `TypeError` 异常。
如果没有抛出该异常，则测试失败。或者抛出了不同的异常，测试也会失败。

在 `test_no_path_raises()` 中，检查了异常类型。
此外，还可以检查异常消息是否正确，或者检查异常的其他方面，例如额外的参数：

```Python
# ch2/test_exceptions.py
def test_raise_with_info():
    match_regex = 'missing 1.* positional arguemnt'
    with pytest.raises(TypeError, match=match_regex):
        cards.CardsDB()

def tset_raises_with_info_alt():
    with pytest.raises(TypeError) as exc_info:
        cards.CardsDB()
    excepted = 'missing 1 required posistional argument'
    assert excepted in str(exc_into)
```

`match` 参数是一个正则表达式，并将异常输出进行匹配。
也可以使用 `exc_info` 或其他变量名来检查异常是否包含额外参数，特别是当它是一个自定义异常时。
`exc_info` 对象将是 `ExceptionInfo` 类型。
这种交错的断言创建的测试很难调试和维护，因为后来的开发者无法知道原始的测试意图。
坚持使用 Given-When-Then 或 Arrange-Act-Assert 模式，可以使测试保持专注，并提高测试的可维护性。

这种三阶段结构是这里使用的测试函数风格。

```Python
from cards import Card

def test_to_dict():
    # GIVEN a Card object with known contents
    c1 = Card('something', 'brain', 'todo', 123)

    # WHEN we call to_dict() on the object
    c2 = c1.to_dict()

    # THEN the result will be a dictionary with known content
    c2_excepted = {
        'summary': 'something',
        'owner': 'brain',
        'state': 'todo',
        'id': 123,
    }
    assert c2 == c2_excepted
```

- Given/Arrange - 开始状态：设置数据或环境为下一步动作做准备

- When/Act - 已执行某些动作：这是测试的重点，我们希望能正常工作的行为

- Then/Assert - 预期的结果或结束状态：在测试的最后，取保动作产生了期望的行为

这样做的想法是，保持测试函数的组织性，并聚焦于测试单个行为。
该结构也能帮组理解其他测试用例。
专注于一个起始状态有助于思考其他可能相关的状态，以便通过相同的操作进行测试。
同样的，专注于结果的思想有助于思考其他可能的结果，例如测试失败或错误情况，这些应该通过其他测试用例测试。

### Grouping Tests with Classes

pytest 还支持通过类来组织测试：

```Python
        c2 = Card('something', 'brian', 'todo', 123)
        assert c1 == c2

        c1 = Card('something', 'brian', 'todo', 123)
        c2 = Card('something', 'brian', 'todo', 4567)

    def test_inequality(self):
        c1 = Card('something', 'brain', 'todo', 123)
        c2 = Card('completely different', 'okken', 'done', 123)
```

### Running a Subtest of Tests

pytest 可以通过以下的方式来运行测试子集

| Subtest                       | Syntax                                             |
| :---------------------------- | :------------------------------------------------- |
| Single test method            | pytest path/test_module.py::TestClass::test_method |
| All tests in a classes        | pytest path/test_module.py::TestClass              |

| Tests by marker               | ...                                                |

## pytest Fixtures

Fixtures 夹具是 pytest 在测试函数之前运行的函数。
可以在 fixture 中运行任何希望的代码，可以使用 fixtures 来测试提供一个数据集以供操作。
可以使用 fixtures 将系统置于已知状态，fixtures 也用于为多个测试获取数据。

### Getting Started with Fixtures

下面先来看一个 fixture 的例子展示测试函数和 fixture 如何一起使用的：

```Python
# ch3/test_fixtures.py
import pytest

@pytest.fixture
def some_data():
    """Return answer to ultimate question."""
    return 42

def test_some_data(some_data):
    """Use fixture return value in a test."""
    assert some_data == 42
```

这里的 `@pytest.fixture()` 装饰器用于告诉 pytest 这个函数是一个 fixture。
当将 fixture 名称放到函数的参数列表中去，pytest 就会知道使用 fixture 并运行它。
fixture 可以用于实现一些功能，也可以向测试函数返回数据。

在 Django 社区，fixture 通常指应用初始化的时候要加载到数据库中的一些数据。
无论是那种意思，测试 fixture 具体的是 pytest 提供的一种机制，它允许测试函数中的 "getting ready for" 和 "cleaning up after" 代码分离出来。

pytest 在处理 fixture 异常时和测试函数不一样。
在测试函数中的异常会产生 "Fail" 结果，而在 fixture 中的异常汇报为 "ERROR"。
这种区别有助于识别为什么测试没有通过。

pytest fixtures 是一种独特的机制，让它在众多框架中脱颖而出。

### Using Fixtures for Setup and Teardown

```Python
from pathlib import Path
from tempfile import TemporaryDirectory
import cards

def test_empty():
    with TemporaryDirectory() as db_dir:
        db_path = Path(db_dir)
        # database object
        db = cards.CardsDB(db_path)

        count = db.count()
        db.close()
        assert count == 0
```

这段测试代码首先获取到数据库 url，然后创建一个数据库对象，再从数据库中获取到数量。
可以看到代码很乱，我们可以使用 fixture 来组织更好的代码：

```Python
# ch3/test_count.py
import pytest

@pytest.fixture()
def cards_db():
    with TemporaryDirectory() as db_dir:
        db_path = Path(db_dir)
        db = cards.CardsDB(db_path)
        yield db
        db.close()

def test_empty(cards_db):
    assert cards_db.count() == 0
```

这样写马上就能看到，测试函数简单了不少。
因为这里将所有数据库初始化的代码放到了名为 `cards_db` 的 `fixture` 中。

`cards_db` fixture 会为数据库测试准备好。
该函数会 yield 数据库连接对象，当测试运行后，会再关闭连接。
这样就保证了，无论发送无论测试函数中发生什么，连接总是会被关闭。

在这个示例，yield 语句位于 TemporaryDirectory 的上下文管理器 with 块内。
该目录在 fixture 使用和测试运行期间存在。
测试完成后，会将控制权传递回 fixture，然后运行 `db.close()`，最后改块执行完成并清理目录。

要记住：pytest 会检查测试中参数的特定名称，然后寻找和 fixture 一样的参数名。
我们从来不会直接调用 fixture 函数，pytest 会为我们做这件事。

并可以在多个测试中使用 fixture:

```Python
# ch3/test_count.py
def test_two(cards_db):
    cards_db.add_card(cards.Card('first'))
    cards_db.add_card(cards.Card('second'))
    assert cards_db.count() == 2
```

`test_two()` 使用同一个 `cards_db` fixture。
他们向空数据库中添加两个卡片，然后测试数量。

编写 fixture 的时候，函数命名应该准确反应其完成的工作，这样有助于代码的可读性。

### Specifying Fixture Scope

每个 fixture 都有一定的作用域，定义了每个使用该 fixture 的测试函数，启动和关闭的顺序。
该作用域决定了当多个测试函数使用它时，启动和关闭的运行频率。

默认的作用域是 function scope，这意味着 fixture 在每个需要运行它的函数前启动。
类似的，在每个测试结束后，运行关闭部分。

然而，你可能不希望这样。
也许连接数据库非常耗时，或你在生产一个很大的数据，或者从很慢的设备接受数据。
因此我们可能想要修改 fixture 的作用域 scope：

```Python
# ch3/test_mode_scope.py
from pathlib import Path
from tempfile import TemporaryDirectory
import cards
import pytest

@pytest.fixture(scope="module")
def cards_db():
    with TemporaryDirectory() as db_dir:
        db_path = Path(db_dir)
        db = cards.CardsDB(db_path)
        yield db
        db.close()

def test_empty(cards_db):
    assert cards_db.count() == 0

def test_non_empty(cards_db):
    cards_db.add_card(cards.Card("first"))
    cards_db.add_card(cards.Card("second"))
    assert cards_db.count() == 2
```

再测试可以看到 fixture 只运行了一次

fixture 装饰器还允许很多其他参数：

- `scope='function'`
  每个函数运行一次。默认作用域，每个测试函数前调用，函数执行完成后关闭。

- `scope='class'`
  每个类运行一次，无论类中有多少个测试方法。

- `scope='module'`
  每个模块运行一次

- `scope='package'`
  每个包或测试目录运行一次

- `scope='session'`
  每个 session 运行一次

Scope 是和 fixture 一起定义的，而不是调用的地方设置的。
在测试模块中的定义的 fixture，`session` 和 `package` 作用域和 `module` 作用域的行为一样。
威力使用这些其他作用域，需要将他们放入一个 `conftest.py` 文件中。

### Sharing Fixtures through conftest.py

实际上，还可以将 fixtures 放到独立的测试文件中，并且在多个测试文件中共享 fixtures。
需要使用一个 `conftest.py` 文件，该文件可以位于使用它的测试文件所在的同一目录下，也可以位于某个父目录中。
其实 `conftest.py` 也是可选的，这被 pytest 当成 local plugin，并且可以包含 hook functions 和 fixtures。

现在将 `cards_db` fixture 从 `test_count.py` 移动到 `conftest.py` 中去：

```Python
# ch3/a/conftest.py
from pathlib import Path
from tempfile import TemporaryDirectory
import cards
import pytest

@pytest.fixture(scope='session')
def cards_db():
    '''CardsDB object connect to a temporary database'''
    with TemporaryDirectory() as db_dir:
        db_path = Path(db_dir)
        db = cards.CardsDB(db_path)
        yield db
        db.close()
```

并编写测试文件：

```Python
# ch3/a/test_count.py
import cards

def test_count(cards_db):
    assert cards_db.count() == 0

def test_two(cards_db):
    cards_db.add_card(cards.Card('first'))
    cards_db.add_card(cards.Card('second'))
    assert cards_db.count() == 2
```

运行测试：

```console

        ch3/a/test_count.py::test_count (fixtures used: cards_db).
        ch3/a/test_count.py::test_two (fixtures used: cards_db).
TEARDOWN S cards_db

=================== 2 passed in 0.01s ============================
```

fixture 只能依赖相同或范围更广的 fixture。
因此 function-scope fixture 可以依赖其他 function-scope fixtures，或者 class-, module-, 和 session-fixtures。

> 不要 Import conftest.py, pytest 自己会做这件事

### Finding Where Fixtures Are Defined

现在已经将 fixture 从测试模块移出到 `conftest.py` 文件了。
可以在测试目录的每个层面都使用 `conftest.py` 文件。
测试可以使用与测试函数同一层的任何 fixture，或者任何级别的父目录中，直到项目根目录。
